# Vibe Check: Testing the Security and Maintainability of LLM-Generated Vibe Code

> **Important:** Do **not** run any generated code files from the `Outputs`, `Stale_Outputs`, or `Invalid_Outputs` folders on your machine.  
> These files are produced by Large Language Models and may contain insecure, broken, or unsafe code.  
> Only the pipeline scripts inside the `Pipeline` folder are intended to be executed.

---

## Overview

This project presents a reproducible benchmarking pipeline for evaluating the **security**, **maintainability**, and **validity** of Python code generated by Large Language Models (LLMs).

The pipeline was developed as part of a final year Computer Science project at ATU Donegal. Its purpose is to compare how different LLMs respond to the same programming prompts and to assess the quality of their generated code using static analysis and structured scoring.

The system automatically:

- Reads a set of predefined prompts.
- Sends each prompt to multiple LLM providers.
- Saves the generated Python code in organised output folders.
- Checks whether each generated file is syntactically valid.
- Runs static analysis using SonarQube.
- Exports analysis findings into CSV files.
- Produces structured datasets for comparison and dashboard reporting.

The generated code is **never executed**. It is treated as static text and analysed only through syntax validation and static analysis tools.

---

## Project Aim

The aim of this project is to investigate whether LLM-generated Python code differs across models in terms of:

- Valid output rate.
- Number of static analysis issues.
- Security-related findings.
- Maintainability issues.
- Code smells.
- Frequency and type of invalid outputs.
- Practical suitability for controlled benchmarking.

This allows different models to be compared in a consistent and repeatable way.

---

## Key Features

- Automated multi-model code generation pipeline.
- Supports comparison between Gemini, ChatGPT, Claude, Claude Opus, and GitHub Models.
- Timestamped output storage for traceability.
- Archiving of previous runs for reproducibility.
- Syntax validation using Python AST parsing.
- Static analysis using SonarQube.
- SonarQube issue export through the Sonar API.
- CSV-based scoring and analysis.
- Dashboard-ready output files.
- Designed for safe analysis without executing generated code.

How the Pipeline Works
1. Prompt Loading

The pipeline reads predefined prompt files from the Prompts folder.

Each prompt is used consistently across all supported models so that the generated outputs can be compared fairly.

2. Code Generation

Each prompt is sent to the configured LLM providers.

The current pipeline supports:

Gemini
ChatGPT
Claude
Claude Opus
GitHub Models

Each model is asked to return Python code only. The response is saved as an app.py file inside the relevant model output folder.

3. Output Storage

Generated files are saved using a timestamped naming format.

Example:

Outputs/Gemini/prompt1/2026-04-24_034557_app.py

This structure makes it possible to trace:

Which model generated the file.
Which prompt was used.
When the output was created.
Whether the file was valid or invalid.
Which issues were detected by static analysis.
4. Syntax Validation

Before any static analysis results are interpreted, each generated Python file is checked using Python AST parsing.

This checks whether the generated file is syntactically valid.

If a file is invalid:

It is not executed.
It is recorded in scoresheet.csv.
It may be moved to Invalid_Outputs.
It is excluded from some valid-output comparisons.

This step is important because LLMs do not always produce runnable or complete code.

5. Static Analysis with SonarQube

Valid generated files are scanned using SonarQube.

SonarQube is used to detect:

Bugs.
Code smells.
Maintainability issues.
Some security-related concerns.
Rule violations.

The scan is run against the generated outputs, not against the core pipeline code.

6. Issue Export

After SonarQube analysis, detected issues are exported using the SonarQube API.

The exported data is saved to:

Scoring/sonar_issues.csv

Each row represents an individual detected issue.

Example columns include:

timestamp, model, output_file, issue_type, severity, rule, message, file, line, status
7. Analysis and Reporting

The exported CSV files can be analysed to produce summary metrics such as:

Total outputs per model.
Valid outputs per model.
Valid output rate.
Total issue count.
Issues per valid file.
Severity breakdown.
Issue type by model.
Combined model comparison metrics.

These outputs are used for the project dashboard and final report evaluation.

Setup Instructions
1. Clone the Repository
git clone <repository-url>
cd vibe-pipeline
2. Create a Virtual Environment
python3 -m venv .venv
source .venv/bin/activate

On Windows:

python -m venv .venv
.venv\Scripts\activate
3. Install Dependencies
pip install -r requirements.txt
4. Configure Environment Variables

Create a .env file in the root of the project.

Example:

GEMINI_API_KEY=your_gemini_key_here
OPENAI_API_KEY=your_openai_key_here
ANTHROPIC_API_KEY=your_anthropic_key_here
GITHUB_TOKEN=your_github_models_token_here
SONAR_TOKEN=your_sonarqube_token_here
SONAR_LOGIN=your_sonarqube_token_here

The exact variables required depend on which providers are enabled in the pipeline.

Do not commit the .env file to version control.

5. Start SonarQube

SonarQube must be running locally before static analysis can be completed.

Default local address:

http://localhost:9000

If using Docker, SonarQube can be started separately before running the pipeline.

6. Run the Pipeline

From the project root, run:

python Pipeline/run_pipeline.py

Only run the main pipeline script.

Do not manually run generated files from the output folders.

Important Safety Warning

This repository intentionally stores code generated by LLMs.

That code may contain:

Security vulnerabilities.
Hardcoded placeholder secrets.
Broken logic.
Poor validation.
Unsafe patterns.
Syntax errors.
Misleading comments.
Incomplete implementations.

Generated code files are included only as research artefacts for static analysis.

They should not be executed, reused, deployed, or treated as safe examples.

Output Files
Scoring/scoresheet.csv

This file records the result of each generation attempt.

It may include:

timestamp, model, output_file, status, syntax_valid, notes

This file is useful for checking whether each model successfully produced usable Python code.

Scoring/sonar_issues.csv

This file stores exported SonarQube findings.

It may include:

timestamp, model, output_file, issue_type, severity, rule, message, file, line, status

This file is used to compare issue counts and issue types across models.

Scoring/analysis/combined_metrics.csv

This file provides a higher-level comparison of model performance.

Typical metrics include:

model, total, valid, valid_rate, issue_count, issues_per_valid_file

This is one of the main files used for evaluation and dashboard reporting.

Example Metrics

Example combined metric structure:

model,total,valid,valid_rate,issue_count,issues_per_valid_file
chatgpt,44,44,1.0,3,0.068
claude,43,39,0.907,16,0.410
gemini,44,43,0.977,5,0.116

These values are produced from pipeline runs and may change depending on the prompts, models, API availability, and analysis configuration.

Limitations

This project has several important limitations:

SonarQube does not detect all possible Python security vulnerabilities.
Static analysis cannot prove that generated code is safe.
Generated code is not executed, so runtime behaviour is not evaluated.
API rate limits can affect dataset completeness.
Some models may fail to respond or produce invalid code.
Prompt wording can influence output quality.
GitHub Models was used as a practical API-based alternative for controlled comparison.
Results are based on the selected prompt set and should not be treated as universal model rankings.
